Working with CIs

 

Create a CI
1. While logged into the Service Desk Console as a Configuration Manager, open the Configuration Item workspace. A list of configuration items appears.

Sample CI List

2. Choose a template from the New drop-down list. For example, New Printer. Each template has different values in various fields. The System provides many out of the box templates for different CI types. In addition to the common attributes that are required across all CI types, these CI templates accommodate CI details of varying complexity. Typical fields could include unique identifier, type, name, description, version, location, owner, status and others depending on the type of CI. The following screen shots show examples of different required fields for the different CI types.
3. Enter information into the fields.
common fields

These fields are common to all templates.

Field Description
Status

Choose from the drop-down list. The default values are:

  • Planned. Draft state; prior to the design state.
  • Design. Work in progress; service is being designed.
  • Transition. Design phase complete; preparing to update the status to Production.
  • Production. Service is in use.
  • Retired. Service is no longer in use. See Retire a CI below.
Name Enter a unique name for the configuration item.
CI Version Enter the version of the configuration item.

Create a CI from an Incident

If the incident reports a configuration item failure, you can search for that configuration item and link it to the incident record.

However, if you need to create a new CI (including service) that is associated with the incident you can also do so from the incident record. The CI is linked to the incident after you create it. You can also view the CIs that the customer currently owns.

Linking helps Problem Management to identify which items may be potential problems.

1. Within the incident record, click the CI tab, and then click New > <CI>.
2. Enter information only into the required fields. Required information is indicated by a red asterisk (*).
3. Click Save. Discovery services populate most of the fields after scanning and auditing the item.

The CI is linked to the incident and appears in the CI tab of the incident record.

Retire a CI

When a CI is decommissioned or disposed, set the CI status to retired and attach any supporting documentation.

Delete a CI

Deleting CIs is independent of deleting escalation exceptions unless they are linked.

Deleting escalation exceptions used by escalation watches isn't advisable since the escalations will stop working for certain incidents or Service Requests in progress. Ensure the escalation exception that you want to deleted isn't being used currently by a watch that is still in Run, Pause, or Reset & Run states.

When deleting services, one of the places impacted would be the Service Catalog, and the application references services throughout various parts. Also, some initialization rules referencing existing Service records, such as Service Desk, are impacted.

Remember, escalation schedules are hierarchical. The escalation schedule at the top level of the hierarchy should never be deleted. Also, default (non-SLA) schedules underneath the top-level Escalation Schedule should not be deleted. Otherwise, the escalation engine will not process the incident or service request, based on a non-SLA based schedule.

  • Within the Service Desk Console, in the CI workspace, select the item you want to delete, then click Delete from the toolbar.
